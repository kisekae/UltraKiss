KiSS Configuration (CNF) File

 

A KISS set is composed of the following files:

* Image file (CEL)

  • This is the image data to be moved. Image files can be of different types such as Cel, GIF, JPG, PNG, BMP, and PPM
  • A cel file is an array of pixels. Each pixel identifies the color of a particular point on the screen.
  • Depending on the image file format pixel colors can be specified in the file or through indexes into a palette file.
  • With cel files using external palettes, pixel code 0 is always a transparent pixel.

* Palette file (KCF)

  • This is a table where each image pixel is an index into the table that defines the actual color of pixels.
  • Palette files can contain many different tables or palette groups, also known as multipalettes.
  • Palette color index 0 is always assumed to be a transparent color.

* Configuration file (CNF)

  • This file describes the overlapping order and positions of cels.
  • The KISS viewer reads this first and uses this file to identify and read other files.
  • This file is a text file created with a text editor.
  • This file may be changed by KISS program's save function.

 

 

Titles and Comments

If the first character in a KiSS Configuration file is a semicolon ';' then the complete text line is considered to be a comment. The line is not, in general, interpreted by the KiSS viewer. Exceptions to this rule exist for FKiSS statements that begin with ';@' and other directive statements such as ;INCLUDE and other viewer specific extension commands.

It is a common practice for the first line in a KiSS CNF file to identify the program that created the file.

Example:

; Kisekae UltraKiss configuration file created on Sun Jan 06 10:22:53 MST 2008
; FKiSS code compatibility level is 5

 

Include Files

The ;INCLUDE directive was introduced to KiSS with the FKiSS 4 enhancements. This directive is intended to facilitate automatic inclusion of required files to the KiSS configuration.

For example, expansion sets add new images or files to the KiSS configurations where new images and configurations can replace existing entities of the same name. Expansion files also contain new CNF files with FKiSS code that reference the new images. With early viewers the base file had to be loaded into the viewer and expansion files had to be entered into the configuration through a viewer specific command. If the ;INCLUDE directive is specified in the expansion file then the required base files can be automatically loaded concurrently with the expansion file.

Syntax:     ;INCLUDE filename1, filename2, ...

Each filename must specify a relative or absolute path to a KiSS archive file. File names must be delimited by quotes (") if the file name contains spaces. Relative paths are relative to the current CNF location. The ;INCLUDE directive is case sensitive and must be specified exactly as shown with capital letters and no space after the semicolon. One or more ;INCLUDE directives can be specified in a CNF file.

If multiple ;INCLUDE directives are used they should be ordered in the sequence in which the files should be loaded. If multiple filenames are specified on one ;INCLUDE directive then the filenames should also be ordered in the sequence in which the files should be loaded.

 

Example:

;INCLUDE    bd13ks.lzh

 

Option Section

The ;[Option Section] directive was introduced by UltraKiss to allow for the specification of viewer specific options that are required for correct set operation. The ;[Option Section]directive is optional. If used, it is case sensitive and must be specified exactly as shown and with no space after the semicolon.

Option settings are either true or false, or the actual option value. Each option setting must appear on its own line following the [Option Section] heading and the line must begin with a semicolon.

 

Syntax:     ;[Option Section]

Screen Section

The ;[Screen Section] directive was introduced by UltraKiss to centralize the specification of set specific options for playfield size and border colors. This directive is not required and it exists simply for structural organization. In general, KiSS screen specifications can appear anywhere within a CNF file. The set options are specified through specific control directives as described below.

Syntax:     =memoryK
       (width,height)
       [integer

Column 1 char
Purpose
Example
=
 memory in kilobytes =800K
(
 playfield size in pixel width and height (<horizontal size>,<vertical size>)
[
 border color palette index into palette file 0, or an integer RGB value [12

 

Example:

;[Screen Section]
[2
(600,380)

 

Palette Section

The ;[Palette Section] directive was introduced by UltraKiss to centralize the specification of set specific palette file specifications. This directive is not required and it exists simply for structural organization. In general, KiSS palette file specifications can appear anywhere within a CNF file.

Syntax:     %filename

The first palette file in the CNF is used by the border color specification found in the Screen Section. Palette files are numbered in the sequence in which they appear in the CNF beginning from number 0.

A KiSS KCF palette file can contain an unlimited number of palette groups. Each palette group is an independent table of colors that can be used by any palette type image cel.

Palette file specifications must begin with a '%' character in column 1. The filename specification may be proceeded by one or more spaces and can be a fully qualified or relative path name to a file on the system. Filenames that contain spaces should be delimited by quotes (").

 

Example:

;[Palette Section]
%doll.kcf              ; palette file 0
%clothes.kcf           ; palette file 1
%clothesg.kcf          ; palette file 2
%doll_g.kcf            ; palette file 3
%ikg.kcf               ; palette file 4

 

Cel Section

The ;[Cel Section] directive was introduced by UltraKiss to centralize the specification of set specific cel file specifications. This directive is not required and it exists simply for structural organization. In general, KiSS cel file specifications can appear anywhere within a CNF file. Images overlay one another according to the order in which they appear in the CNF. Images that appear first in the file are drawn on top of images that appear later in the file. Cel files are internally numbered in the sequence in which they appear in the CNF beginning from number 0. Therefore, lower numbered cel files are drawn on top of higher numbered files.

Cel specifications must begin with a '#' character in column 1. The specification serves to identify the objects in the KiSS set and the images within each object. The specification also identifies the palette file associated with the image and the viewer pages on wich the image can be seen.

Syntax:     #<Object>[.<Fix>] <Cel filename> [*<Palette number>[.<Palette group >]] [:<Page Set number>...]
                        [; [<%Tag>...] [<!Cel group [: <frame ...>] ...] comment]

The object number is defined through the cel specification. More than one image cel specification line can have the same object number. This is how more than one image is associated to a unique object.

The optional fix specification is an integer number that defines the lock count for the image cel. A deficiency with KiSS is that lock restrictions actually apply to objects and not images. Therefore, the object lock count is established to be the maximum lock count of all image cels associated to the same object number.

The cel filename specification must be proceeded by one or more spaces and can be a fully qualified or relative path name to a file on the system. Filenames that contain spaces should be delimited by quotes (").

The palette number should relate to one of the palette files specified in the Palette Section. Image cels that do not use a palette do not need to specify a palette number and any such specification will be ignored. Image cels that are associated to an unknown palette number may not be drawn properly.

The optional palette group specification is an UltraKiss extension to identify the fixed palette group of colors that will be applied to the image. This fixed palette group will not change as color sets change and is related to the setpal() and letpal() FKiSS action commands.

The optional page set numbers should be delimited by spaces. If no page sets are defined then the image cel is visible on all page sets. If the page set colon {:) prefix is specified without any following page numbers then the image cel is not visible on any page. If a single multidigit sequence of undelimited numbers is used then this single number is considered to be a set of page specifiers where each digit specifies a page from 0 to 9. This simplification is provided for compatibility with early KiSS sets where viewers did not support more than 10 pages or color sets.

The optional tag specifier is an early extension introduced to configure transparency and other specifiers for the image cel. All tag specifiers must be delimited by one or more spaces and must proceed any cel group specifications and comment text.

The optional cel group specifier is an FKiSS 4 extension implemented to allow for additional groupings of image cels beyond movable objects. Every image cel can be a member of an unlimited number of groups where each group has a unique name. The image cel can also be assigned a unique frame number which assists with identifying a subset of images within the group.

 

Example:

;[Cel Section]
#21.100 ikg.cel       *4  : 0 7               ;
#20     warning.cel                           ;
#6      coat_1.cel    *1  : 2                 ;
#10     blazer1g.cel  *2  : 1 3 4 6 7         ;
#5      blazer_1.cel  *1  : 1 2 3 5 7         ;
#13     blazer1p.cel                          ;
#12     blouse1g.cel  *2  : 0 1 3 4 5 6 7     ;

 

Tags

Tags are optional specifications that appear at the end of a cel specification line as comment specifiers. The semicolon that begins the comment is required. Some viewers will require that there be no spaces between the semicolon and the first tag specifier, although this is not strictly necessary. One or more spaces are required between each different tag to delimit the tag tokens.

 

%u (unmap tag)

Tag Syntax:             ; %u

The unmap tag sets the initial visibility state of the image to unmapped.

Example:

#20     warning.cel                           ; %u %t64 %g

 

%t (transparency tag)

Tag Syntax:             ; %tN

The transparency tag sets the initial transparency of the image. A transparency N value of 0 is fully opague and 255 is fully transparent. There can be no spaces between the transparency flag %t and the first digit of the number N.

Example:

#20     warning.cel                           ; %u %t64 %g

 

%g (ghost tag)

Tag Syntax:             ; %g

The ghost tag sets an initial ghost state for the image. A ghosted image is unresponsive to mouse clicks.

Example:

#20     warning.cel                           ; %u %t64 %g

 

%c (transparent color tag)

Tag Syntax:             ; %cN

The color tag is used to set an initial transparent color for image cels that do not maintain transparency. The color value N is an integer representation of the RGB color components, where the red, green, and blue color values range from 0 to 255 inclusive. There can be no spaces between the color flag %c and the first digit of the number N.

Example:

#7.32767 one.gif : 1 ;%c16777116


%size

Tag Syntax:             ; %size[width,height]

The size tag sets an initial size for the image, for those images or other components that may not maintain sizing information. The %size tag requires integer width and height specifiers contained within a list as shown. Each integer is a measure in pixel size for the image. The square brackets must be specified. There can be no spaces between the size flag %size and the square bracket '[' that begins the width and height list.

Example:

#1.9999 label1.label ;%size[470,50] %attributes[text="FKiSS5",fc=16711680]


%offset

Tag Syntax:             ; %offset[x,y]

The offset tag sets an initial offset for the image, for those images or other components that may not maintain offset information. The %offset tag requires integer X and Y specifiers contained within a list as shown. Each integer is a measure in pixel size for the image. The square brackets must be specified. There can be no spaces between the size flag %offset and the square bracket '[' that begins the offset list. Both positive and negative numbers can be specified for the X and Y offset values.

Example:

#2.9999 string.textfield ;%size[327,26] %offset[108,0] %attributes[text=""]

 

%attributes

Tag Syntax:             ; %attributes[attribute1,attribute2,...]

The attribute tag is used to specify optional attributes for component type image cels, such as background colors, initial text fonts, and so on. Each attribute is generally unique for the component. The square brackets must be specified. There can be no spaces between the attribute flag %attributes and the square bracket '[' that begins the attribute list. The attribute values are specific to the attribute and can be of text or numeric types. Text attributes must be delimited by quotation marks. Attribute terms must be specified in a comma separated list.

Example:

#7.9999 label7.label ;%size[400,26] %attributes[text="",border=1,borderstyle=line]

 

! (cel group tag)

Tag Syntax:             ; !<name> [: n1 n2 n3]

The celgroup tag is the cel group name starting with an exclamation mark, and optionally a frame definition which is a colon followed by frame numbers. As the example below shows, one cel instance can be assigned to many different cel groups. Cel group definitions can also be placed on a new line as long as the line definitions are preceded by a semicolon. The examples show two equivalent representations for the specification of cels 'three.cel' and 'four.cel' . An image cel can have multiple frames and many different image cels in the same group can have the same frame number.

Example:

#10.100 three.cel : 0 ; !Number :3 
;                       !Card :1 
;                       !Queen
#10.100 four.cel  : 0 ; !Number :4 
;                       !Card :1 
;                       !Jack

Example:

#10.100 zero.cel  : 0 ; !Number :0 !Card :0
#10.100 one.cel   : 0 ; !Number :1 !Card :1 !Ace
#10.100 two.cel   : 0 ; !Number :2 !Card :1 !King 
#10.100 three.cel : 0 ; !Number :3 !Card :1 !Queen
#10.100 four.cel  : 0 ; !Number :4 !Card :1 !Jack
#10.100 five.cel  : 0 ; !Number :5 !Card :2 !Ten
#10.100 six.cel   : 0 ; !Number :6 !Card :3 !Nine
#10.100 seven.cel : 0 ; !Number :7 !Card :4 !Eight

;@ press(!Card)           ; a press event on any cel in a cel group
;@ unmap(!Card)           ; unmap all images in a cel group
;@ setframe(!Card,5)      ; map frame 5 in the cel group
;@ let(A,"!Ace")          ; assign a string to a variable 
;@ map(A)                 ; map the cel group !Ace
;@ let(A,"Number")        ; assign a string to a variable 
;@ map(!A)                ; map the cel group !Number

 

Page Set Section

The ;[Page Set Section] directive was introduced by UltraKiss to centralize the specification of object page positioning specifications. This directive is not required and it exists simply for structural organization. In general, KiSS page set specifications can appear anywhere within a CNF file.

Syntax:     $<Palette group> [<x,y> <x,y> ...]

The palette group specifier is an integer number from 0 to N, where N is the maximum number of palette groups or color sets taken over all palette files. This is the initial color setting that is applied to all palette cels when the page is first opened. For cels that reference a palette file that does not contain the required palette group the image cel color is not changed.

The optional list of x,y pairs represent coordinate positions for each object in the set, beginning from object number 0. If a coordinate pair does not exist for a given object number then the object is positioned at location (0,0).

Each $ page set specification applies to one page in the order in which the $ lines appear in the CNF file. The first $ line applies to page 0, the next to page 1, and so on. There is no limit as to the number of pages that can be defined in this way. Page set specifications can be continued on the following line or lines providing that each successive line in the specification begins with a blank, or space character in column 1.

Object coordinates pairs are specified with a '*', for object numbers that do not exist in the configuration or are not visible on the page.

 

Example:

;[Page Set Section]
; Page 0
$0 * 0,0 * * * * * * * * * * * * 130,25 * * * * * * *
; Page 1
$0 * 0,0 130,60 560,300 577,96 619,331 605,331 591,331 113,115 0,0
   0,0 590,188 590,155 590,122 * * * * * * * *

 

FKiSS Section

The ;[FKiSS Section] directive was introduced by UltraKiss to centralize the specification of FKiSS code. This directive is not required and it exists simply for structural organization. In general, FKiSS code must appear together and at the end of a CNF file. The ;[FKiSS Section]directive is almost always followed by the ;@EventHandler line that begins FKiSS code.

Example:

;[FKiSS Section]
;@EventHandler

;@initialize() ;
;@ let(MaxFrames,7) ; the number of images in a block
;@ let(Magic,6) ; the magic frame image in a block
;@ random(MusicFile,0,13) ; choose a random music file
;@ random(BackFile,0,4) ; choose a random background file

; Unmap all main menu hint text strings.

;@ unmap("slidertxt.gif") ; unmap slider game text

 

; **********************************************************************
; Main Menu Event Section.
; ======================================================================

; The set(0) event initializes for the main menu screen.

;@set(0) ;
;@ music("") ; stop music
;@ letset(Set) ; retain our current page
;@ gosub(NewBackground) ; choose a new background
;@ gosub(DestroyObjects) ; destroy all prior objects
