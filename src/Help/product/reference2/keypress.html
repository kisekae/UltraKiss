keypress() 

Syntax:     keypress (keyliteral)

This event is triggered by pressing the key with the specified letter or number. There is no autorepeat and no difference between uppercase and lowercase characters. Cursor keys are specified as "left", "right", "up" and "down". The key literal should be enclosed within double quotation marks.

FKiSS 4 introduced keyboard control, but it is limited and incomplete. FKiSS 4 only allows keypress() events for the digits 0 through 9 and the alphabetic characters A through  Z. Note that there is no means to determine case.  A special exception was made to allow keypress() events for cursor control keys.

FKiSS 5 extends the keypress() event by recognizing all keys on the keyboard. Non-display characters such as Shift, Alt, Ctrl, Backspace, function keys and so on are specified with their mnemonic name. The table below provides the names for all keys on a standard IBM keyboard. 

Note that FKiSS 5 keypress() events remain case insensitive. This is because multi-key combinations such as Shift-A generate two keypress() events. One is for the Shift key and the other is for the letter A. For case sensitive entry of keyboard events, FKiSS 5 introduced the keytype() event, This event fires on mouse release and properly recognizes key modifiers such as Shift, Ctrl, and Alt. To obtain the actual case sensitive key pressed and modifier keys in use, FKiSS 5 introduced the letkeychar() and letkeymodifier() action commands. These commands set string variables to the appropriate values.

Specifying keypress() or keyrelease() events for key combinations that are typically used by the viewer should override viewer standard action. This allows all keys to be processed by the KiSS application.

More than one keypress() event can be specified for a specific key. If sequential event processing is enforced each event will fire in the sequence in which it was defined, otherwise they are queued for concurrent processing. 

Note that there is no defined order for event processing in KiSS. Single threaded viewers that do not support concurrent events will typically process the keypress() event before the keyrelease() and keytype() events. However, for multithreaded viewers that support concurrent event processing there is no guarantee that the keypress() event will complete before the keyrelease() event begins. No assumptions should be made for the sequence in which events are processed.

Related events: keyrelease(), keytype()
Related actions: letkey(), letkeymodifier(), letkeymap(), leykeychar(), letkeycode(), letkeystring()

 

Compatibility:

FKiSS 4 allows numeric and alphabetic keys keypress ("a"), keypress("0") Triggered when the key is pressed on the keyboard.
FKiSS 5 allows all keys.  keypress ("Num Lock")  or  keypress ("Space") Triggered when the key is pressed on the keyboard.

 

UltraKiss Options:

KeyCase enable or disable key case recognition default value "true" (FKiSS option)
RetainKey  active key press is required for letkey() and letkeymap() actions  default value "true" (Compatibility option)

;@keypress("a") ; 
;@ notify("A key pressed") ;
